01

The problem Syntax solves

Starting a website from nothing sounds simple until the same setup decisions appear again: typography, spacing, mobile behavior, themes, metadata, testing, local previewing, accessibility, and deployment.

Syntax handles that repeatable foundation. It generates ordinary HTML, CSS, and optional JavaScript that belong to your project immediately. You still design the site. You simply do not rebuild the plumbing every time.

Syntax should handle

  • A readable responsive foundation
  • Project metadata and a manifest
  • Selected theme and navigation behavior
  • Basic accessibility and browser tests
  • Local preview and GitHub Pages preparation

Your project should own

  • The content and information architecture
  • The visual identity and art direction
  • Project-specific layouts and interactions
  • Images, illustrations, and editorial decisions
  • Anything too specific to belong in every site

02

Where Syntax fits

Syntax is most useful for focused sites that should be polished without becoming technically heavy.

Content site

An article, guide, reference collection, or small publication.

Portfolio

A personal or studio site with selected work and a clear contact path.

Product page

A landing page for an app, tool, service, experiment, or early product.

Small interactive project

A calculator, game, visualization, or utility that needs a dependable shell.

Start from scratch instead when the project is truly disposable, is only a few lines of HTML, or already requires a specific application framework for a concrete reason.

03

The two-folder model

The current private workflow uses two neighboring folders. One contains Syntax itself. The other is your actual site.

~/Projects/
├── syntax/   ← generator and framework workshop
└── olive/    ← your clean website

You run setup from syntax and tell it to write into the empty project folder. After generation, nearly all everyday work happens in your project folder.

Do not use “Use this template” for a clean consumer project. It copies the full Syntax workshop—Component Lab, internal tests, recipes, tooling, and documentation. Create an empty repository instead.

04

Your first project, step by step

1. Create an empty GitHub repository

Create it without a README, license, .gitignore, or repository template.

2. Create a projects folder

mkdir -p ~/Projects
cd ~/Projects

3. Clone Syntax and your empty repository

git clone https://github.com/stokewell/syntax.git
git clone https://github.com/YOUR-USERNAME/YOUR-PROJECT.git

A warning that the new repository is empty is expected.

4. Install the Syntax generator

cd ~/Projects/syntax
npm install

5. Generate your project

npm run setup -- --output ../YOUR-PROJECT

The two dots mean “go up one folder.” This works because the two folders sit beside each other.

6. Answer the setup questions

Press Enter to accept the value in parentheses. Enter recipe IDs in lowercase exactly as shown.

Project name: Olive

Recipe [blank/portfolio] (blank): blank

Visual direction [editorial/product] (editorial): editorial

Canonical URL or “not yet”: not yet

Deployment [none/github-pages-root] (github-pages-root): press Enter

A button destination must be a URL, path, filename, or page fragment. Use #ornamental, not Ornamental.

7. Enter the generated project

cd ../YOUR-PROJECT
ls -la

8. Install project development tools

npm install
npx playwright install chromium

9. Test and preview

npm test
npm run serve

Open http://localhost:4173/ unless Terminal reports another address.

05

Terminal commands without the mystery

pwd
Shows which folder Terminal is currently in.
cd ~/Projects/olive
Moves directly to a known folder from anywhere.
cd ..
Moves up one folder.
ls -la
Lists files, including hidden files such as .git.
Control + C
Stops a local server in the current Terminal window.
The % or $ before commands is the Terminal prompt. Do not type it.

06

Testing and previewing

npm test starts a temporary server and checks metadata, links, mobile overflow, keyboard behavior, selected features, and major accessibility violations.

npm run serve starts a server that stays open for manual inspection.

Using another port

npx http-server . -p 4174 -c-1

Then visit http://localhost:4174/. Tests may still expect their configured port, so stop a conflicting server before running npm test or update both configurations together.

07

Saving the baseline to GitHub

git add .
git commit -m "Initialize project with Syntax"
git push origin main
  • git add . stages the files.
  • git commit creates a local snapshot.
  • git push uploads it to GitHub.

08

Publishing with GitHub Pages

  1. Push the project to GitHub.
  2. Open Settings → Pages.
  3. Choose Deploy from a branch.
  4. Select main and /(root).
  5. Save and wait for deployment.
https://YOUR-USERNAME.github.io/YOUR-REPOSITORY/

When the site is nearly ready, preview release checks:

npm run prepare:ship

After reviewing the report:

npm run prepare:ship -- --write

09

Common problems we already stepped on

“Recipe must be blank or portfolio”

Enter the lowercase ID: blank, not Blank.

“Destination must be a valid URL, path, or fragment”

Use #ornamental, article.html, /about/, or a full URL.

Tests load the wrong website

Another server may already use the port. Stop it with Control + C, or find it with:

lsof -nP -iTCP:4173 -sTCP:LISTEN
EADDRINUSE

The port is occupied. Stop the old process or preview on another port.

cd ../olive says the folder does not exist

Relative paths depend on your current folder. Run pwd, then use cd ~/Projects/olive.

There is a .DS_Store file

It is a macOS Finder file. Remove it and add .DS_Store to .gitignore.

10

What happens after setup?

Work inside your generated project. Replace the starter content. Edit index.html, site.css, site.js, and your assets. Keep what helps. Delete or override what does not serve the real site.

Syntax is successful when the finished site feels completely like your project—and you barely notice the starter underneath it.
